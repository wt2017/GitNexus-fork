1. Architecture Overview

GitNexus has three components that work together:

ComponentPurposeRuntime
CLI / Indexer Parses source code, builds knowledge graph, stores in .gitnexus/ Node.js, local shell
MCP Server REQUIRED Exposes tools (query, impact, context, etc.) to Claude Code via stdio stdio subprocess spawned by Claude Code
HTTP API OPTIONAL REST API on port 4747; used by Web UI and third-party clients npx gitnexus serve or Docker

Data flow:

Your Code Repo
    ├── src/...
    └── .gitnexus/          <-- index stored here (LadybugDB)

Claude Code
    └─spawn──> npx gitnexus mcp
                  └─reads──> ~/.gitnexus/registry.json  <-- global repo registry
                  └─opens──> <repo>/.gitnexus/lbug      <-- per-repo graph DB
Key principle: The MCP server is stateless and read-only (for queries). Index creation is always done via the CLI. The MCP server discovers repos through the global registry.

3. Installation

Option A: Global Install (Recommended)

npm install -g gitnexus

This pins gitnexus to PATH, giving the fastest MCP startup (avoids cold npx cache delays).

Option B: Per-Project Install

cd /path/to/your-project
npm install --save-dev gitnexus

Option C: npx (One-shot, slower startup)

npx gitnexus@latest --help
Using npx for MCP may trigger installs on cold cache. Claude Code’s default MCP timeout is ~30s; if the install exceeds this, the MCP server fails to start. Global install avoids this.

Quick Setup (Auto-detect Editors)

gitnexus setup

This auto-detects Claude Code, Cursor, Codex, and OpenCode; writes global MCP configs. You can still do it manually below.

4. Indexing Your First Repository

Navigate to your project root and run the indexer.
cd /path/to/your-project
gitnexus analyze

This performs a full 12-phase pipeline:

  1. Scan — file paths and sizes
  2. Structure — folder/file tree
  3. Parse — tree-sitter AST extraction (symbol nodes, CALLS/IMPORTS edges)
  4. Routes / Tools / ORM — framework-specific extraction
  5. Cross-file — type propagation in import order
  6. MRO — method override resolution
  7. Communities — Leiden algorithm clustering
  8. Processes — execution flow tracing
  9. Persist — load graph into LadybugDB
  10. FTS indexes — BM25 full-text search
  11. Embeddings — optional semantic vectors (local Snowflake model, no API calls)
  12. Context files — generate AGENTS.md / CLAUDE.md
No LLM API calls during analyze. The entire pipeline is deterministic local computation. The only step that uses an external API is gitnexus wiki, which is a separate command.

Incremental Updates

gitnexus analyze           # incremental (default)
gitnexus analyze --force   # full rebuild
gitnexus analyze --embeddings  # also generate/update semantic vectors

What Gets Created

your-project/
├── src/...
└── .gitnexus/                  <-- gitignored automatically
    ├── lbug                    <-- LadybugDB graph database
    ├── lbug.wal
    ├── lbug.lock
    ├── meta.json               <-- index metadata (lastCommit, stats)
    ├── parse-cache/            <-- content-addressed parse cache shards
    └── ...

The global registry is created at:

~/.gitnexus/registry.json       <-- maps repo names to absolute paths

5. Claude Code MCP Configuration

Step 1: Add MCP Server

Run one of the following based on your install method:

# A. Global install (fastest startup -- recommended)
claude mcp add gitnexus -- gitnexus mcp

# B. npx fallback (slower cold start)
claude mcp add gitnexus -- npx -y gitnexus@latest mcp

# C. Project-local install
claude mcp add gitnexus -- npx gitnexus mcp

Or manually edit ~/.claude/settings.json:

{
  "mcpServers": {
    "gitnexus": {
      "command": "gitnexus",
      "args": ["mcp"]
    }
  }
}

Step 2: Restart Claude Code

Exit and restart claude. The MCP server starts automatically. You should see in the first message:

GitNexus: MCP server starting { repoCount: 1, repos: ["your-project"] }

How MCP Works

When Claude Code starts:

  1. Spawns gitnexus mcp as a stdio subprocess
  2. The subprocess reads ~/.gitnexus/registry.json to discover repos
  3. Opens LadybugDB connections lazily on first query (5-min idle eviction, max 5 concurrent)
Do not run gitnexus mcp and gitnexus serve simultaneously on the same repo. They compete for the LadybugDB file lock (.gitnexus/lbug.lock), which can corrupt the index or cause query failures.

6. Installing Agent Skills

Skills are workflow guides stored under .claude/skills/ (global or project-local). They tell Claude Code when and how to use GitNexus tools.

Standard Skills (4 built-in)

SkillWhen to Use
gitnexus-exploringUnderstanding architecture, unfamiliar code, “How does X work?”
gitnexus-debuggingTracing bugs, errors, “Why is X failing?”
gitnexus-impact-analysisBefore editing code, “What will break?”
gitnexus-refactoringRename, extract, split, safe restructuring

Install Globally (All Projects)

mkdir -p ~/.claude/skills
git clone https://github.com/abhigyanpatwari/gitnexus.git /tmp/gitnexus-src
cp -r /tmp/gitnexus-src/.claude/skills/gitnexus/* ~/.claude/skills/
rm -rf /tmp/gitnexus-src

Install Per-Project

mkdir -p .claude/skills
cp -r /path/to/gitnexus-repo/.claude/skills/gitnexus/* .claude/skills/

Auto-Install via analyze

Running gitnexus analyze (without --skip-skills) automatically installs standard skills to the project’s .claude/skills/ and generates repo-specific skills under .claude/skills/generated/ if you use --skills.

7. Optional: Claude Code Hooks

Hooks provide automated behaviors: session-start context injection and post-commit staleness detection.

SessionStart Hook

Automatically injects GitNexus context when a session starts in an indexed repo.

Edit ~/.claude/settings.json:

{
  "hooks": {
    "SessionStart": [
      {
        "command": "bash /path/to/gitnexus/hooks/claude/session-start.sh",
        "description": "Inject GitNexus context on session start"
      }
    ]
  }
}

The hook script is in the GitNexus repo at gitnexus/hooks/claude/session-start.sh.

PostToolUse Hook (Staleness Check)

After git commit or git merge, prompts the agent to re-index if the codebase has drifted.

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "node /path/to/gitnexus/hooks/claude/gitnexus-hook.cjs",
        "description": "GitNexus staleness check after git operations"
      }
    ]
  }
}
The hook does not run analyze automatically (to avoid blocking the agent for minutes). It only notifies the agent, which then asks you if you want to re-index.

9. Daily Usage Workflow

Before Editing Code

# 1. Impact analysis (MANDATORY)
gitnexus impact "MyFunction" --direction upstream

# Or via Claude Code MCP:
gitnexus_impact({target: "MyFunction", direction: "upstream"})

Before Committing

# 2. Detect changes
gitnexus detect-changes --scope staged

# Or via MCP:
gitnexus_detect_changes({scope: "staged"})

Safe Rename

# 3. Dry-run first
gitnexus rename "oldName" "newName" --dry-run

# 4. Apply if satisfied
gitnexus rename "oldName" "newName"

Keep Index Fresh

# After major changes
gitnexus analyze

# Or full rebuild if suspect corruption
gitnexus analyze --force

10. Docker Deployment (HTTP API Mode)

For teams or CI environments, you can deploy the HTTP API via Docker. This does not replace MCP; it provides a backend for the Web UI and third-party HTTP clients.

One-Command Setup

git clone https://github.com/abhigyanpatwari/gitnexus.git
cd gitnexus
docker compose up -d

This starts:

  • Server on http://localhost:4747 (HTTP API + indexer)
  • Web UI on http://localhost:4173

Indexing inside the Container

# Mount your host workspace
WORKSPACE_DIR=$HOME/code docker compose up -d

# Register an already-indexed repo (read-only mount)
docker compose exec gitnexus-server gitnexus index /workspace/my-repo

# Or clone and index inside the container
docker compose exec gitnexus-server bash
git clone https://github.com/org/repo /data/gitnexus/repo
cd /data/gitnexus/repo && gitnexus analyze && gitnexus index .

Docker Key Facts

ItemValue
Imagesghcr.io/abhigyanpatwari/gitnexus:latest (server), ghcr.io/abhigyanpatwari/gitnexus-web:latest (UI)
Named volumegitnexus-data persists registry + cloned repos at /data/gitnexus
Workspace mount/workspace:ro (read-only) — only for registering pre-built indexes
EnvGITNEXUS_HOME=/data/gitnexus moves the global registry inside the container
SignatureImages are Cosign-signed; verify before production use
_stdio MCP is not containerized in the official setup. The Docker images run gitnexus serve (HTTP), not gitnexus mcp (stdio). If you need MCP in a container, you must solve path mapping, stdout isolation, and LadybugDB file-locking manually.

11. Troubleshooting

SymptomCauseFix
MCP server failed to start Cold npx cache timeout Switch to global install: npm i -g gitnexus; update MCP config to use absolute gitnexus path
Index is stale Code changed since last analyze Run gitnexus analyze in the repo root
Database is locked gitnexus serve and gitnexus mcp running simultaneously Stop one; only run one server mode at a time per repo
No indexed repos yet Registry empty or path mismatch Run gitnexus analyze then gitnexus status
Tree-sitter build fails Missing python3/make/g++ Install build tools, or set GITNEXUS_SKIP_OPTIONAL_GRAMMARS=1
Worker parse timeout Very large files or slow disk gitnexus analyze --worker-timeout 60
High memory usage Large repo + embeddings Skip embeddings: gitnexus analyze --skip-embeddings; increase Node heap

12. Security & Privacy Notes

  • No network calls during indexing. The entire gitnexus analyze pipeline is local. No source code leaves your machine.
  • Embeddings are local. Uses HuggingFace transformers.js with Snowflake arctic-embed-xs (384D). No API key needed.
  • Global registry stores only paths and metadata. ~/.gitnexus/registry.json contains repo paths, lastCommit hashes, and symbol counts — never source code.
  • Index is stored inside the repo. .gitnexus/ is automatically added to .gitignore and .git/info/exclude. Do not commit it.
  • Only gitnexus wiki calls an LLM API. This is an explicit opt-in command requiring an API key. All other tools are fully local.
  • Path traversal guards. All file reads in MCP tools are validated against the registered repo root.
